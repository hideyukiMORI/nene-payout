## ★教訓

- **観測と因果は別物で、因果こそ測る対象だった。** conformance の exit 1 は実際に見た〔観測〕。しかし「なぜ 1 か」を `cli.js` を読まずに「unknown も blocker」と書いた〔因果を測らず〕。前夜の Dependabot「13」（暗算）も同型。exit も本数も観測はしたが、機構と引き算という因果側を測らなかった。**検算は疑ったときにやることではなく、書く前に必ずやること。**
- **渡された「確定」は、自分の binding 規約が最も強い検算器。** 「二重非準拠確定」も「ADR 0019=hooks/」も、payout の terms.md / frontend-standards.md / ADR 本文を読めば違った。指揮側の裁定でも merge 条件でも、前提が現物と食い違えばそのまま書かない。
- **決め打ちせず適用して観測する。** A1 の dry-run `rewrite 0` に対し「壊れる」と予測したが、手作業移設は禁止なので適用して check で観測 → 予測は外れ（正しく rewrite された）。予測を証拠にせず、観測を証拠にする。
- **相互検証は双方向に効く。** 自分のミス（conformance 機構・Dependabot 数）は hub / fleet の裏取りが捕まえ、指揮側の前提誤り（二重非準拠・ADR）は自分の実測が捕まえた。どちらも main に乗る前に止まった。
