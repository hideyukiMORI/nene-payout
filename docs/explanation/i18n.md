# Internationalization (i18n) — Binding Design

**Status: binding.** NeNe Payout is a **bilingual (ja / en) service by premise**.
Every piece of text shown on any screen is managed in **message catalog files**,
never hardcoded, so the language switches smoothly and completely at runtime.

Modeled on the `nene-records` / `nene-invoice` i18n implementation (a small,
dependency-free catalog with a typed English source of truth and per-locale
fallback), adapted to Payout's ja/en scope and the embeddable widget.

See: [`frontend-standards.md`](../development/frontend-standards.md),
[`pages.md`](./pages.md), ADR 0006 (English repository docs),
[ADR 0012](../adr/0012-utc-storage-jst-display.md) (UTC store / JST display).

---

## 1. Principles (binding)

1. **No hardcoded UI text.** Every user-facing string (labels, buttons,
   headings, placeholders, validation/error messages, empty states, toasts,
   aria-labels, the widget) comes from a message catalog via `t(key)`.
2. **English is the typed source of truth.** `messages/en.ts` defines the
   `MessageCatalog` type (the full key set). Other locales are
   `Partial<MessageCatalog>` and fall back to English for any missing key.
3. **Japanese ships complete.** For a release, `ja` must define **every** key in
   `en` (no fallback gaps in shipped UI). A key-parity test enforces this.
4. **Switching is instant and persistent.** Changing locale re-renders from the
   in-memory catalog (no reload), persists to `localStorage`, and updates
   `document.documentElement.lang` / `dir`.
5. **The API is not translated.** Server responses (incl. Problem Details
   `type` / `title`) stay English stable contracts; the UI localizes by mapping
   stable `code` / problem `type` to catalog messages (§6).

---

## 2. Locales

| Locale | Label | Notes |
| --- | --- | --- |
| `ja` | 日本語 | Primary audience (Japan SMB) |
| `en` | English | Typed source of truth + fallback |

- `SupportedLocale = 'ja' | 'en'`.
- `DEFAULT_LOCALE` is resolved at runtime (§5); the **fallback catalog** is
  always `en`.
- Adding a locale is additive: a new `messages/<locale>.ts` (`Partial<MessageCatalog>`)
  plus a `LOCALES` entry — no call-site changes.

---

## 3. File structure (binding)

```
frontend/src/shared/i18n/
  locales.ts            SupportedLocale, LOCALES meta (label, dir), DEFAULT_LOCALE, resolveLocale()
  translate.ts          MessageKey = keyof MessageCatalog, translate() with en fallback + {{param}}
  i18n-context.tsx      I18nProvider (state, persistence, document lang/dir)
  i18n-context-ref.ts   React context + useI18n() hook
  messages/
    en.ts               source of truth — `export const en = { ... } as const`; exports MessageCatalog type
    ja.ts               Partial<MessageCatalog>
    index.ts            getMessages(locale): Partial<MessageCatalog>
    locales.test.ts     key-parity test (§7)
```

This lives under `frontend/src/shared/i18n/` (shared across admin UI and widget).

---

## 4. Key naming & catalog shape (binding)

- Flat string keys, dotted namespaces:
  - `common.*` — shared (actions, fields, errors, dialogs)
  - `admin.{feature}.{element}` — admin screens (features per [`pages.md`](./pages.md): `receivedInvoices`, `vendors`, `payments`, `settings`, `auditLogs`, `users`, `nav`, …)
  - `widget.*` — embeddable payment widget
- Parameter interpolation uses `{{paramName}}`.

```ts
// messages/en.ts (source of truth)
export const en = {
  'common.actions.save': 'Save changes',
  'common.error.validation': 'The submitted data is invalid.',
  'admin.receivedInvoices.pageTitle': 'Received invoices',
  'admin.payments.amountDue': 'Amount due: {{amount}}',
  'widget.pay.submit': 'Pay',
} as const

export type MessageCatalog = typeof en
```

```ts
// messages/ja.ts
import type { MessageCatalog } from './en'
export const ja: Partial<MessageCatalog> = {
  'common.actions.save': '変更を保存',
  'common.error.validation': '入力内容が正しくありません。',
  'admin.receivedInvoices.pageTitle': '受取請求書',
  'admin.payments.amountDue': '支払金額: {{amount}}',
  'widget.pay.submit': '支払う',
}
```

> Message **keys** are content identifiers. Their single source of truth is
> `messages/en.ts` (the typed `MessageCatalog`), **not** `docs/terms.md` — `terms.md`
> governs code/API/DB identifiers (ADR 0017). Keys still follow the naming rule
> above; an unregistered/typo'd key fails type-checking against `MessageCatalog`.

## 5. Switching mechanism (binding)

- `I18nProvider` holds the current locale and exposes `t(key, params?)` via
  `useI18n()`. Components call `t('...')` only — never read catalogs directly.
- On mount, locale is resolved: `localStorage['nene-payout-locale']` →
  `navigator.language` → `DEFAULT_LOCALE` (`resolveLocale()` maps e.g. `ja-JP` → `ja`).
- `setLocale(next)` persists to `localStorage`, updates state (instant re-render),
  and sets `document.documentElement.lang` / `dir`.
- A locale selector is available in the admin shell; the choice survives reloads.

```tsx
const { t, locale, setLocale } = useI18n()
<h1>{t('admin.receivedInvoices.pageTitle')}</h1>
<p>{t('admin.payments.amountDue', { amount: formatMoney(amountCents, locale) })}</p>
```

## 6. Errors & server messages (binding)

- The API returns **English** Problem Details with a stable `type` and a
  machine-readable `code` for validation errors (see
  [`../development/api-error-responses`](../development/nene2-runtime-reference.md)).
- The UI **does not display the server's English text**; it maps the stable
  problem `type` / validation `code` to a `common.error.*` (or field-specific)
  catalog message. Unmapped errors fall back to `common.error.unknown`.
- This keeps the API contract stable and the UI fully localized.

---

## 7. Completeness & quality gates (binding)

- `locales.test.ts` asserts **key parity**: every key in `en` exists in `ja`
  (and any other shipped locale). A missing or extra key fails the test.
- Type safety: `t()` only accepts `keyof MessageCatalog`; unknown keys are a
  TypeScript error, so a typo cannot reach runtime.
- `npm run check --prefix frontend` (type-check + lint + test) must pass — a
  missing `ja` translation is a release blocker (§1.3).

---

## 8. Formatting (binding)

- **Dates/times**: stored UTC; displayed in **JST** (ADR 0012). Format via the
  locale where wording differs; the calendar value is JST.
- **Money**: stored as **integer minimum currency units**; format for display
  only at the view edge (locale-aware), never mutate the integer.
- Pluralization/number formatting use `Intl` helpers where needed; keep
  catalog strings free of concatenation that breaks translation.

---

## 9. Widget i18n (binding)

- The embeddable widget uses the same catalog (`widget.*`). Its locale is set by
  a `data-locale` attribute on the embed (or inherits the host when configured),
  resolved through `resolveLocale()` with `en` fallback.
- No hardcoded strings in widget code; no secrets/PAN (ADR 0010) regardless of
  locale.
